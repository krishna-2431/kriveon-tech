1. Executive Summary
This Technical Requirements Document (TRD) outlines the architectural, structural, and implementation guidelines for the official KriveonTech website. The platform serves as a high-performance, premium digital storefront designed to transition KriveonTech's brand perception to a professional software engineering agency. 
In Phase 1, this application operates strictly as an informational and lead-generation engine without a backend database, user authentication, or data collection forms. The engineering focus is on maximum performance, Search Engine Optimization (SEO), and zero-friction lead generation utilizing direct communication channels (Click-to-Call and Click-to-WhatsApp). 
2. Overall System Architecture
The system utilizes a modern, edge-deployed Jamstack architecture relying on Next.js 15 for Server-Side Rendering (SSR) and Static Site Generation (SSG).

Frontend: Next.js 15 (App Router) with React 19.
Hosting & CDN: Vercel Edge Network for global low-latency access. 
Analytics: Google Analytics 4 and Microsoft Clarity (client-side injection).
Static Assets: Next.js Image optimization (next/image) serving WebP assets conditionally.

6. Component Architecture
All components default to React Server Components (RSC) to reduce client bundle size. The "use client" directive is restricted strictly to components requiring interactivity.
Server Components: Hero, Footer, Timeline, ServiceCards, Layout, content sections.
Client Components: Navbar (for mobile toggle and scroll state), WhatsAppFAB (for hover states and dynamic URL generation), BentoCard (for Framer Motion hover animations). 
Reusable Components Highlights
PrimaryButton: Electric Blue background, white text, glow on hover. 
WhatsAppFAB: Fixed position (bottom right), ARIA-labeled, expands on hover. 
TimelineNode: Circular indicator for the 8-step process, active blue pulse animation. 
7. State Management
Given the lack of user authentication or shopping carts, global state management (e.g., Redux, Zustand) is not required.
Client-Side State: Handled via standard React useState and useContext (if needed for theme/mobile menu).
Component Rendering: Leverage Next.js Server Components. Pages fetch JSON content sequentially on the server and pass plain objects to client components if interaction is required.
8. Content Management Strategy
For Phase 1, a headless CMS is an over-engineering risk. Content will be managed via static local files.
Format: Strongly typed JSON arrays inside the content/ directory.
Images: Visuals and logos are stored in public/assets/ and referenced in JSON.
Future-proofing: The UI components are decoupled from the data layer. Transitioning to a CMS (e.g., Sanity, Contentful) in Phase 2 only requires swapping the JSON import for a fetch call in the page-level Server Component.
9. SEO Architecture
SEO is a primary driver for the targeted personas.
Next.js Metadata: Implementation of Open Graph (og:title, og:image) and Twitter Cards.
Structured Data (JSON-LD):
Organization & LocalBusiness schemas injected into app/layout.tsx.
Service schema injected into /services.
FAQPage schema injected into /faq. 
Robots & Sitemap: Automated generation using app/sitemap.ts and app/robots.ts.
Semantic HTML: Strict adherence to hierarchical <h1-h6> tags, <main>, <article>, and <section> structures. 
10. Performance Strategy
To hit the target of 95+ Mobile/Desktop Lighthouse scores: 
Image Optimization: All raster images must use <Image/> from next/image for WebP conversion and lazy loading. SVGs used for icons (Lucide). 
Font Optimization: next/font/google used to preload Inter/Geist and JetBrains Mono, eliminating FOIT (Flash of Invisible Text). 
Code Splitting: Native to Next.js App router.
Third-party Scripts: Analytics loaded via next/script using strategy="lazyOnload".
11. Accessibility (WCAG 2.1 AA)
Color Contrast: The primary Electric Blue (#007AFF) and dark backgrounds (#0A0A0A to #111111) must pass AA contrast ratios. 
Keyboard Navigation: All CTAs (<a>, <button>) must have explicit :focus-visible ring styling (e.g., ring-2 ring-blue-500 ring-offset-2 ring-offset-background).
ARIA Labels: Mandatory for icon-only elements like the WhatsApp FAB and hamburger menu.

The core conversion mechanism relies on zero-friction, direct contact.

Click-to-Email: mailto: protocol.
15. Analytics & Event Tracking
Given the absence of forms, macro-conversions are measured via click events on contact links. 
Create a custom utility trackEvent(eventName, properties) mapping to GA4 data layer.
Events to track:
click_whatsapp_fab
click_call_header

16. Security
Headers: Implement Next.js headers() in next.config.js to enforce strict Security Headers:
Strict-Transport-Security (HSTS). 
X-Frame-Options (DENY).
X-Content-Type-Options (nosniff).
Content-Security-Policy (CSP) restricting scripts to self and approved analytics domains. 
Data Validation: While no POST forms exist, ensure any future URL parameters are strictly validated to prevent Reflected XSS.
17. Testing Strategy
Linting/Formatting: Strict ESLint and Prettier configs enforced via Husky pre-commit hooks.
Component Testing: React Testing Library for verifying complex UI logic (e.g., FAB expansion).
E2E Testing: Playwright script to verify critical paths:
Navigation loads.
Click-to-call links contain the tel: protocol.
WhatsApp links contain correct wa.me host.
Performance: Automated Lighthouse CI in GitHub Actions targeting 95+ across all metrics. 
18. Deployment Pipeline
Version Control: GitHub repository.
Branching: main (Production), develop (Staging), feature/*.
CI/CD: Vercel automatically deploys preview environments for every Pull Request. Merges to main trigger production deployments. 
Rollback: Vercel instant rollback capability via atomic deployments.
19. Coding Standards
TypeScript: strict mode enabled. No any types permitted.
Naming Conventions:
Components: PascalCase.tsx
Utilities/Hooks: camelCase.ts
CSS/Tailwind: BEM-inspired utility grouping (using clsx and tailwind-merge via a cn() utility).
Import Order: External libraries -> Absolute internal paths -> Relative paths -> CSS/Styles. Enforced via eslint-plugin-simple-import-sort.
20. Future Scalability
Phase 1 establishes a fast, static base. The architecture supports Phase 2 requirements (Client Portal, Admin Dashboard, Payment Gateway) without major refactoring: 
Next.js App Router seamlessly handles adding dynamic loading.tsx and Server Actions for form POSTs.
Authentication (e.g., NextAuth.js/Clerk) can be wrapped via a Provider in layout.tsx for protected routes (/dashboard).
Database integration (e.g., PostgreSQL via Prisma/Drizzle) can be instantiated securely within React Server Components.
21. Risks & Mitigations
Risk
Impact
Mitigation Strategy
Inaccurate Conversion Tracking
Low accuracy tracking ROAS on Google Ads due to lack of a "Thank You" page. 
Implement precise click-event listeners on all tel: and wa.me links via GTM/GA4.
WhatsApp Spam
Exposing raw WhatsApp numbers can invite automated spam. 
Use dynamic messaging templates requiring user intent. Monitor traffic; if severe, consider a basic captcha-gated redirect in Phase 1.5.
Hydration Mismatches
Next.js throws errors if client HTML differs from server HTML.
Avoid relying on window object in initial render without checking useEffect or dynamic imports with ssr: false.
